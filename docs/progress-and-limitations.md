# 当前成果与限制

本文档用于记录 MyClaw 当前阶段已经完成的成果、仍然存在的限制，以及下一步建议。

## 一、本阶段已完成的成果

### 1. 飞书主链路已打通

当前已经完成：

- 飞书长连接接入
- 飞书文本消息接收
- 路由到模型
- 文本回复发送回飞书

这意味着项目已经从“只能本地玩”进入“可以真实放进飞书里使用”的阶段。

### 2. Gateway 命令已可运行

此前 `gateway` 命令只是占位，现在已经可以真正启动：

- Gateway 服务
- 通道管理器
- 飞书通道

### 3. 配置加载方式已完善

现在支持：

- 环境变量指定配置路径
- 用户目录默认配置
- 项目本地配置

这让开发和部署都更方便。

### 4. Agent 回复提取已修复

之前曾出现：

- `(No response)`
- `length` 字段错误

当前运行时已经改成基于 session + 流式事件提取，稳定性明显比之前好。

### 5. OpenRouter 兼容性已改善

已经处理过：

- 某些免费模型无可用端点
- usage 流式统计缺失导致的问题

默认模型也已经切换为更稳的：

- `openrouter/free`

### 6. 飞书助手能力已从纯聊天升级到只读工具辅助

当前飞书不是单纯的大模型问答，而是可以：

- 看文件
- 查文件
- 搜代码
- 看目录

这让它在研发场景下更有实际价值。

## 二、当前能力边界

### 当前擅长

- 研发问答
- 项目结构总结
- 代码检索
- 文件阅读
- 配置解释
- 轻量内容生成

### 当前不适合

- 自动改代码
- 写文件
- 执行命令
- 需要人工审批的高风险工具链
- 长时间自主多步任务

## 三、当前限制

### 1. 飞书通道没有审批闭环

这是现在最重要的限制之一。

当前如果开放高风险工具，可能出现：

- 模型进入工具审批流程
- 但飞书侧没有审批回传
- 导致任务卡住

### 2. Telegram 仍未完成

目前 Telegram 只有基础脚手架，不能视为完整可用。

### 3. `message` 命令尚未落地

当前 CLI 中 `message` 命令还未形成完整功能。

### 4. 文档和代码风格仍有历史残留

项目在快速演进过程中，部分文件仍然存在：

- 注释风格不统一
- 中英文混合
- 旧结构残留

这些问题不影响当前可用性，但后续值得继续整理。

## 四、为什么当前选择“只读工具模式”

不是因为项目做不到更强，而是因为当前阶段更看重：

- 可用
- 稳定
- 不阻塞

对于飞书这种消息通道来说，只读工具是一个很好的平衡点：

- 比纯聊天聪明
- 比全工具安全
- 不容易卡住

## 五、下一步最值得做的事情

### 第一优先级

- 做飞书通道的工具审批回传机制

### 第二优先级

- 支持按通道配置工具集

例如：

- 飞书只读工具
- 本地终端全工具
- 未来 Telegram 只读或受限工具

### 第三优先级

- 补齐 Telegram 通道
- 完成 `message` 命令
- 提升 Gateway 状态输出
- 增加更多诊断日志和排障支持

## 六、当前阶段的结论

MyClaw 目前已经达到“可演示、可试用、可继续扩展”的状态。

它还不是一个功能完备的多平台智能助理产品，但已经具备了非常清晰的下一步演进基础：

- 通道层已经建立
- 路由层已经建立
- 运行时已经建立
- 飞书主链路已经可用

这意味着后续的工作重点不再是“从零开始”，而是“逐步增强能力、补齐边界、提高稳定性”。
